## 弹窗与页面关系
弹窗是用来承载具体详情组件的，也可以用页面或tab来承载这些组件，所以具体组件设计时不要考虑依赖具体的承载方式。
弹窗只定义一个，有单独的 新增、修改、查询组件，且不同数据都使用同一个弹窗，因为不同数据最大区别在于页面本身，而不是展示方式。
弹窗是列表与详情中的承上启下的作用。

## 弹窗定义
定义一个通用的弹窗，分别从crud角度分析
c和rud不同，后者是数据已经存在的情况下进行处理，而c是创建数据。

c：
定义一个泛型弹窗，泛型类型表示新增表单组件，新增时列表弹出此组件

r u d
查看模式和编辑模式是需要切换的,且查看模式时可以直接点击删除。
所以定义个弹窗，包含两个泛型参数，分别对应 编辑表单和查看组件

而d这不需要UI组件，直接在弹窗中处理即可。

注：新增和修改 里面有很多字段，按上面的设计都得分两份做，可以考虑单独定义一个名为：编辑组件 的东东，新增和修改组件引用它即可。
另外c和u其实区别也蛮大，新增往往比修改能填写的数据更多，有些字段是新增时填写后，就不许修改的。

# 概述

BXJG.MudBlazor项目下层调用应用服务接口，上层为具体项目提供抽象组件

最开始的思路是 提供一个 List和Detail，它们之间的跳转由于方式有多种（比如弹窗、页面跳转、或其它）
所以抽象部分做成独立的。

然后在这个基础上提供子类，实现最常用的弹窗方式。

但想想，由于交互方式基本就那几种，若使用继承，要写一堆类来免去泛型参数，所以决定不用继承的方式
而是在抽象组件中写死 if else


一般我们点击新增、修改、查看 按钮会弹出一个窗口，里面显示某个实体的详情。
此时传递些数据过去，比如 新增时把用户在列表中选择的 商品类别传递过去，以便新增时默认类别就是刚才在列表页中选择的类别。
再比如修改时将id传递到弹出中的表单。

用户查看、编辑数据。

点击保存时提交到数据库，也可能编辑失去焦点时立即提交到数据库。

最后提示成功或者失败

成功时弹窗关闭，然后刷新列表页。

注意BXJG.Mudblazor项目是针对本项目 特定于Mudblazor的项目，它下层调用应用服务，上层为具体项目提供公共组件，它与具体业务是无关的。


# 弹窗还是标签还是？
点击新增、修改、查看按钮后，并不一定是通过弹窗来展示实体详情，也可能是多标签的方式，或调整到编辑列表
所以表单组件设计时不应考虑到底用哪种方式，表单组件应该是独立的，可以默认由一种方式展示，比如弹窗。

不过仔细考虑，列表页进入表单页面，无非就：页面跳转、弹窗、tab等方式、或直接隐藏列表页，然后展示表单，由于目前mudblazor中多标签未与路由关联，所以目前不考虑多标签方式。

具体用哪种方式，应该是具体项目来决定的。也不应该由用户觉得，那样用户思想包袱太重。

不同方式对后续传参，进入 退出 有影响的。

由于方式种类比较固定，后续处理可以使用if else方式来处理，没必要搞抽象，那样不够简单，直观。

# 数据传递
不同展示方式传递数据的方式不同，若是弹窗应该使用MudBlazor中弹窗传值方式。
若是页面跳转方式进入表单页面，则通过路由参数传递
或者其它方式。

# 展示模式
通常有新增、修改、查看、自由模式。
新增和修改：所有录入框可编辑
查看时 是直接展示数据，木有输入框
自由：进入时 是只能查看，但鼠标移动到某项时变成可编辑，编辑完成后立即提交。

站在抽象的角度，抽象的表单组件，它只关心是新增 还是修改，因为新增时 组件加载时，它啥都不干就行，修改时需要通过id去查询数据，
查看时也需要去查询数据，自由模式是针对修改的场景，所有抽象中只关心是一并提交，还是每个属性变化时都提交。
按用户习惯来说，他们已经属性了 批量编辑之后，点击提交按钮
