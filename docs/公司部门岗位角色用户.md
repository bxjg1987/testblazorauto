# 部门
abp默认只有个组织单位，但在我们的租赁系统中，发现客户服务平台中，不同客户需要维护自己的部门信息，客户的部门和 我们自己公司的部门字段可能不同。
所以最终设计为 各自建立自己的部门，都继承OrganizationUnit

# 岗位
指业务上的岗位职责
操作工多半不需要登陆系统，那么他就不与权限挂钩。
财务，是个岗位，财务人员需要登陆系统，就需要与权限挂钩。

# 角色
主要用来划分权限。

# 是否合并岗位和角色
abp默认只有角色，木有岗位。
按概念来说应该分开，比如“喷漆工”，他都不登陆系统，分配角色就没意义。
但反过来说，把张三的岗位设置成“出纳”，如果不合并，还必须建立个“出纳”的角色，然后把张三分配到“出纳”角色，
这种情况系统用户就觉得奇怪了，为什么建立了岗位还要建立角色，还分配两次。

按abp的设计，就是把角色和岗位混一起的，所以 部门下面有多个岗位也在abp中体现了，即一个组织单位下可以关联多个角色，不过它设计的是多对多的关系。

如果合并，开发起来会简单很多。

最终决定合并，也搞成继承的，形式，PostEntity继承Role

# 用户

原本abp有个User类，表示系统用户。

但往往一个系统中存在多种类型的用户，比如学校系统，有学生、家长、老师、教务等，这些用户的字段不同
登陆界面不同
管理界面不同
业务处理不同

在我们的项目中是归结到不同的应用中。

我们是通过ef继承来实现不同用户的，还是单表，不过每种用户一个子类。

目前没有对户主用户做继承，因为目前还没有开发户主的功能，后续需要单独定义个子类，表示户主。数据迁移也暂时注释了户主的用户和角色。

StaffInfoEntity 员工，继承User，  系统后台的用户就是此用户。
CustomerStaffInfoEntity 客户那边的员工，继承User
