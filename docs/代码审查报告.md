### 1.3 客户数据过滤器逻辑缺陷 ✅ 已修复

**问题描述：**
ZLJDbContext中客户过滤器的表达式使用了连续的`==`比较运算符，导致在特定条件下过滤器行为与预期相反——不过滤时反而返回不属于当前客户的数据。

**涉及文件：**
- `src\ZLJ.EntityFrameworkCore\EntityFrameworkCore\ZLJDbContext.cs` 第137行

**关键代码：**
```csharp
Expression<Func<TEntity, bool>> mayHaveOUFilter = e => ((IMustHaveCustomer)e).CustomerId == CurrentCustomerId == IsCustomerFilterEnabled;
```

**原因分析：**
表达式 `A == B == C` 的实际执行逻辑是 `(A == B) == C`。当 IsCustomerFilterEnabled 为 false 时，条件变为 `CustomerId != CurrentCustomerId`，即返回所有不属于当前客户的数据，而不是返回所有数据。虽然ABP框架在 ShouldFilterEntity 返回 false 时不会应用此过滤器，但如果 ShouldFilterEntity 返回 true 而 IsCustomerFilterEnabled 为 false（例如配置不一致），就会出现数据泄露。

注意：代码下方有被注释掉的正确实现（第143-151行），说明开发者曾意识到这个问题但采用了当前这种有风险的写法。

**建议处理：**
- 恢复使用被注释掉的正确实现方式：先判断 IsCustomerFilterEnabled，再构建过滤表达式
- 添加单元测试验证过滤器在各种条件下的行为

**处理结果：已修复。** 恢复了被注释掉的正确实现方式：在if条件中先判断IsCustomerFilterEnabled，过滤表达式仅保留 `CustomerId == CurrentCustomerId`，避免了连续`==`运算符导致的逻辑反转问题。

**涉及文件：**
- `src\ZLJ.EntityFrameworkCore\EntityFrameworkCore\EFMap\BaseInfo\StaffInfoMap.cs` 第15行

**关键代码：**
```csharp
builder.Property(x => x.IdNumber).HasColumnType("varchar").HasMaxLength(ZLJ.Core.Share.ZLJConsts.StaffInfoIdNumberMaxLength);
```

**原因分析：**
HasColumnType("varchar")生成的SQL类型是varchar不带长度，SQL Server默认为varchar(1)，后续的HasMaxLength可能无法正确覆盖。正确写法应该是将长度包含在列类型中。

**建议处理：**
- 改为：`builder.Property(x => x.IdNumber).HasColumnType($"varchar({ZLJ.Core.Share.ZLJConsts.StaffInfoIdNumberMaxLength})")`
- 或仅使用：`builder.Property(x => x.IdNumber).HasMaxLength(ZLJ.Core.Share.ZLJConsts.StaffInfoIdNumberMaxLength).IsUnicode(false)`

**处理结果：已修复。** 移除了可能有问题的HasColumnType("varchar")调用，改用HasMaxLength配合IsUnicode(false)，由EF Core自动推断为varchar(18)。

### 2.12 BXJGUtilsDIExt.GetKeyedServiceBXJG无任何防护 ✅ 已忽略

**问题描述：**
GetKeyedServiceBXJG方法中存在多重风险：字典访问可能抛出KeyNotFoundException，GetRequiredService可能抛出InvalidOperationException，强制类型转换可能失败，均无任何防护。

**涉及文件：**
- `src\Modules\Utils\BXJG.Utils.Share\Extensions\BXJGUtilsDIExt.cs` 第34-38行

**关键代码：**
```csharp
public static TService GetKeyedServiceBXJG<TService>(this IServiceProvider keyedService, string key) 
{

    return (TService)keyedService.GetRequiredService(t);
}
```

**原因分析：**

- 添加类型转换的安全检查
- 考虑使用GetService替代GetRequiredService，返回null而非抛异常

**处理结果：已忽略。** 该方法是为了解决ABP不支持Keyed Service的临时方案，key未注册是编程错误，应在开发阶段发现，不需运行时防护。
