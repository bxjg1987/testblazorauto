# 是否依赖abp
考虑过 应用服务接口 + 应用服务默认实现 + storage接口 + storage默认实现（ef） 的方式来实现进销存模块，
不依赖abp。这样基本的进销存功能可以作为一个库使用

完全可以，不过会增加工作量，为了适配需要定义， 设置接口、通知接口、事件接口等，
基本上需要定义abp提供的所有基础设施的接口。


# 是否剥离主数据到单独的模块中

进销存功能中的，产品、产品分类、供应商、客户、仓库 等信息可以视为主数据，它可能在很多其它模块之间共享。

进销存模块还可以细分为 进、销、存、主数据 等多个单独的模块，
若每个模块都分离出来，需要建一堆项目，感觉挺麻烦。

换个思路，将进销存业务功能放一个模块，主数据单独放一个模块？
我们的目标是实现简单的，基本的进销存，如果主数据放单独的模块，
实现的效果是有个复杂的系统，可能原本的进销存模块不够通用，只想使用主数据，而自己实现进销存。

或嫌进销存功能太多了，只想使用库存模块

我们的目标是实现简单的进销存功能，主数据会非常简单，基本就是基础数据的crud，
单独玻璃出来的意义不大，如果嫌不方便，就完全可以自己实现，而不是引用进销存模块的主数据。

由于功能简单，将来剥离也行，所以现在可以尽量将主数据功能定义在自己的文件夹中。

主数据

商品是共用的，供应商和客户可能是同一个单位、所以可以共用为 往来单位
此外进和销都依赖存功能，可以直接引用，反过来的不要

# 是否使用泛型抽象
不要，考虑abp.io的思路，要开箱即用。
主程序需要扩展，使用继承、json、efcore影子属性、动态属性 等方式实现扩展，推荐：继承的方式

泛型抽象，泛型会爆炸

# 我们的框架严重依赖ef
所以core项目中可以引用ef，因为领域层要借助ef查询实现更灵活 更高级的功能。
若仅仅依赖abp提供的仓储就会失去这种能力。
