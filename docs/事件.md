# 后端触发前端事件

业务数据不需要
业务数据crud后，没必要主动推送到前端，因为前端啥时候用就啥时候发起请求就行了。

通知
需要后端推送，这个是abp自带的

前端缓存的数据
如全局配置，前端启动时获取到本地缓存，后端变化后应该通知前端刷新。
不要直接推送过去，这样前端单个数据更新逻辑太复杂了。

通知与缓存数据推送的区别
通知是会持久化的，在线人数少时直接推送的，人多的话，是用的后台作业发送的。
缓存数据的推送无需持久化，仅推送给在线用户。

推送要区分应用、租户、和用户
由于推送是涉及到远程io，某些更改可能不需要全部在线用户都推送
比如改了某个租户的某些配置，这个时候应该只推送给指定租户的在线用户，而不是所有在线用户。

推送延时覆盖
有时候可能在短时间改多个数据，若每个数据都推送比较浪费。
比如 有多个用户，短时间内，更改了多个配项，此时会推送多次。比较浪费。
可以使用延迟覆盖执行，确保指定事件延时10秒，后再统一触发一次事件。延时时间范围内有新的请求时，之前的等待会被取消，新的请求重新记时。
注意，要设置超时，若某些状态一直在改，延迟覆盖执行会导致一直延迟，可能很久才真正触发事件。

前端无需触发后端的事件
前端要调用后端直接走api请求即可。

哪些是跟随事件，哪些是执行时指定
事件名称、事件参数、默认触发范围（应用、租户、用户）、覆盖执行器。
这个几个东东哪些是预定义的，哪些是触发时指定的
考虑下，每个事件都是系统预定义的，所以事件名称代表一个事件，一定是预定义的。
事件参数，肯定是触发时指定的。
触发范围：一个事件对应一个功能，比如：配置变更事件，这个可以触发时，根据情况指定，
也可以预定义3个事件，应用配置更改事件、租户配置更改事件和用户配置更改事件。根据经验，后面的方式更好。
若是户主更改指定租户的配置，则触发时需要指定目标租户id，
若是租户管理员更改自己租户的配置，则触发时不用指定，执行时从IAbpSession中获取，
用户也是一样的道理。
覆盖执行器，这个必须是预定义的。不然每次创建达不到效果。
可以在ioc容器中注入键控服务，注意在abp中需要手动解析，目前的用的windsor不支持net8的键控服务。


## 实现
既然是前端事件，就应该定义在应用层。公共的定义在common中，独特的定义在各自的应用层中。

定义事件常量，或参数类型作为事件名，定义在前后端都可以引用的项目中，也就是 应用层.Share中
定义键控单例服务，注册覆盖执行器。定义在具体的应用层中。

定义前端事件触发器。
	定义应用范围的事件触发方法：事件名称或参数类型、事件参数、传入覆盖执行器的参数
	内部根据事件名称从容器获取覆盖执行器，若有则执行它，否则直接触发前端事件。

	定义租户级别的事件触发方法，参数：时间名或参数类型、事件参数、指定租户id（若不指定自动取当前租户）、覆盖执行器参数

	用户一样

abp的事件
它一个是数据库层面，工作单元提交前后会触发事件，这种事件往往是启动配置时注册事件处理器。
另一种是动态的，运行时 注册、注销、触发事件，
这非常好。
但目前，8.3.1版本中，运行时动态注册事件会报线程安全错误，就是注册事件的同时，可能有事件在触发，abp没有处理线程安全问题，此错误规划到下一个版本会修复。

MediatR
这是个轻量级的事件总线库，貌似不能动态的随时 注册、注销，没仔细研究了。
