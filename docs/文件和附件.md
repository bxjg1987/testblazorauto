# 概述
整个系统中很多功能都会用到文件和附件功能，所以考虑设计一个通用的文件附件模块，任何业务系统都可以引入此模块快速实现文件附件管理。
核心思路如下：
1. 提供全局统一的临时文件上传接口，它接收文件并存储到临时目录中。 定时任务删除临时目录中的文件
1. 新增业务实体时，在具体业务的应用服务中保存实体的同时，将文件移动到正式目录，并在全局统一文件表存储此文件信息，然后使用全局统一的附件表记录实体与文件的关联关系
实体、文件、附件等数据库记录，与具体的物理文件保持在一个事务中
1. 修改时，删除老的文件、插入新文件记录，同时存储新的物理文件
1. 删除时，删除物理文件和文件、附件记录，并在同一个事务中删除业务实体
1. 查询时，每个业务的应用服务提供附件下载接口，它返回文件的流、文件名、mime等信息，因为需要判断权限所以每个应依服务提供此接口

目前只考虑本地存储文件，木有考虑oss的情况。
目前也没考虑大文件分片上传的实现。

# 文件
有一个统一的文件表存储文件信息，一个文件对应一行记录，数据库记录中存储文件的相对路径，加上配置的目录将来可以组合成物理路径。
在设置的目录中存储着物理文件，

FileEntity是数据库实体

BXJG.Utils.Web中定义统一的控制器，它负责临时文件上传，权限控制为登录用户。

文件管理器FileManager负责文件的crud等操作，具体来说它负责将文件存储到临时目录、将文件从临时目录移动到正式目录，维护文件的数据库记录，支持物理文件的删除等。

定时任务属于用例，所以定义在utils的应用层的，但worker是现成，只能在具体应用中去注册
周期性定时任务为：RemoveUploadFileWorker，它在ZLJ.Web.Host模块中注册，它负责删除过期的临时文件。

# 附件
将一个和多个文件与某个信息关联，比如：报销功能中，一个报销单可能关联一个或多个附件
这种关系是直接关联关系，

附件管理器AttachmentManager提供业务实体与文件的关系维护

# 设置
涉及到两种配置，一种是基本永远不变的，比如：物理文件的存储路径
另一种配置是通过abp的settings来实现的，如：允许上传的文件类型，单个文件的大小限制
以及可以考虑多租户中，允许租户的文件数量或总文件大小等。

# 权限
目前仅考虑功能权限，不考虑数据权限。

上传临时文件时是用的统一控制器接收的，它只有登录用户即可上传。

具体的应用服务中做最终上传，此时权限是随具体业务功能的，比如：上传工单关联的照片时，则是在修改和新增工单时直接调用上传完成方法，且无需进一步做权限判断。

删除时往往是删除实体时，再删除关联附件的

重点是下载，可以考虑每个具体的应用服务提供各自的附件下载接口，这样权限非常好控制，但需要所有需要附件的地方都定义对应方法。

也可以考虑统一定义附件下载接口，此时权限就不好控制了，需要用一个权限表记录 哪个实体类型，那几个权限名可以下载，这个表如何维护也比较麻烦，可以不用表，直接用全局静态数据，或者用abp的 模块配置功能
也可以用settings功能，这样在下载时提供权限名称和文件id才能下载，这样前端比较麻烦，但调用具体应用服务的接口也存在一个区分问题，所以是一样的，
相对来说有settings更好，有缓存，不至于占用太多内存

最终决定各应用服务自己搞
