# 硬编码
以前，或者说abp默认的方式
在列表页查询时都是根据所管理的信息定死那几个字段作为条件，查询表达式也是硬编码写。
比如：学生档案管理中，按 学号 考生号 姓名 身份证号 等模糊查询，而出生年月、身高、体重可以按范围查询，或者还包含枚举类型和bool类型的条件
这种方式的缺点如下：
1. 类似的代码很多，写起来繁琐。
1. 不灵活，要调整条件要改条件dto定义，还得修改查询的linq


# 动态条件
前端提交一个条件列表，列表中的每一项都是一个条件，包含：属性名、比较方式、比较值，
后端根据此条件动态拼接linq表达式。

system.linq.dynamic.core支持用字符串方式拼接linq

注意动态条件不处理逻辑表达式，**多个条件之间只有and关系**，因为包含逻辑表达式实现起来太复杂了，用户用起来也复杂.

# 实现
由于动态条件和linq拼接都是通用功能，与abp无关的，因此条件定义和linq处理都定义在BXJG.Common项目中

## CompareType
所有的比较方式用此枚举定义

## CompareTypeMap
哪些c#基元类型，可以使用哪些比较方式，这种映射关系定义在这个类中。
由于这个功能是死的，因此直接定义为静态类。
将来这个映射关系可以提供给前端，用户选择不同字段时，可选的比较方式才能匹配。
>枚举、可空情况需要特别处理。

## ConditionFieldDefine
上面只是定义了比较方式，这里是定义条件，属性名、比较方式、值，每个实例就是一个条件，多个条件之间会使用and查询。

## linq扩展
LinqExt提供扩展方法来简化动态条件的应用

## 关键字模糊查询
这个不属于动态条件的范畴，比如查询的信息包含多个字符串，如果用上面的动态条件，学生姓名 包含 xxx，并且学号包含xxx，这是and关系。
而关键字还是保留原来的硬编码，你通常应该使用 学生姓名包含xxx 或者 学号包含xxx。
