## 粘性会话


api有实时通知，如果仅配置为websocket和禁用传输方式协商，则不需要粘性会话。
最终使用此方式，所以目前是么用开启粘性会话的

特殊的url需要开启时，应该单独定义路由，以免影响原来的实现。



为了不改变单体的配置，还是决定在yarp中支持粘性会话。
https://learn.microsoft.com/zh-cn/aspnet/core/fundamentals/servers/yarp/session-affinity?view=aspnetcore-9.0

无论是客户端的blzor server还是后端api的基于signalR的实时通知，
在分布式环境都需要粘性会话和redis背板，

基本原理是 ：
在轮询和sse场景中（猜测的）
signalR首次连接时，会获得一个cookie，后续请求会携带它，后端的yarp需要配置
基于cookie的粘性会话，它会根据粘性会话值将连接到原始服务器


signalR 连接一般是两次，第一次请求是协商，到底用websocket、sse、轮询？中的哪种？
服务器客户端确认后，客户端再次发起连接请求，

在负载均衡场景中，两次请求可能被路由到不同的服务器了，
所以需要粘性会话，

默认粘性会话cookie比较容易处理，但是wasm的前端又不允许提交cookie

所以想实现就必须换成haader的粘性会话方式，

目前使用此方式：  
一种办法是 直接使用websocket + 跳过协商，
反正是长连接，一旦连接成功就不断了，所以负载均衡不影响
此方式就限定了浏览器端必须支持websocket，其实现在大部分都支持了，
问题不大。


若非要考虑sse 轮询兼容，那就必须协商。

新增一个路由，单独对于signalR处理，
并使用header粘性会话，
前端存储粘性会话key，后续请求时搞到header里。

signalR支持在请求头中设置值吗？
为什么我看到很多地方连身份验证的token都是用querystring传递的，
而不是使用请求头，
是不是说明signalR不支持请求头？

此外粘性会话经过测试，设备实例推送数据时基本上都会推送到某一台

# 后台作业

尽量使用abp的后台作业，不要直接使用hangfire的api

后台作业是有开关的，所以可以仅保留一台服务器执行后台作业，
但这不太好，因为可能刚好配置启用后台作业的服务器挂了，
最好是后台作业使用分布式锁，确保后台作业只有一个执行。

另一方面，若使用hangfire，貌似它可以确保只有一个线程的任务被执行。

但为了避免对hangfire的依赖，还是用abp的后台任务api，所以还是得假定整体后台作业没有hangfire的能力。

# 通知中心
单体情况下，abp已经支持基于signalR的实时通知接口，也支持同时添加多种通知。
要么使用redis signalR背板 ，

简单的情况下，signalR配置为仅支持websoket，关闭传输协商，这样就不需要在yarp做粘性会话。


另一种方式是单独做个通知中心的服务，然后改造下abp增加新的实时通知实现

这个独立的通知服务可以设计为同时支持多种通知，跟abp的类似，比如：邮件、短信、signalR等

这样其实是微服务的方式了，这个通知服务可以给到其它项目中


邮件、短信等的通知是内部后台周期性任务去发送的，
基于signalR的话，需要做身份验证，这个只要两边的jwt密钥配置一样就行， 就是在abp这边登录，在通知中心那边直接解析即可。
或者使用ids4 或 identity connection

授权就更不是问题了。

# 设置
在使用abp的setting时，它内部使用缓存，默认是滑动过期的。
所以通过settingmanager接口获取缓存应该始终是最新的。

重点是我们定义了.net的配置提供器，以从abp setting是中获取应用程序级别的配置。
在进程内使用事件，可以拿到最新的配置，
但A服务器修改配置，B服务器的.net IConfiguration是无法更新最新配置的。
最简单粗暴的是轮询从数据库获取配置

获取配置不要从内存中拿，而是从settingmanager去拿。
这里注意，配置提供器不容易使用ioc，所以可以用IocManager的静态，创建范围后去拿

这样一来，的配置提供器也没必要监控配置变更了。

# 验证码
