## 粘性会话
api有实时通知，如果仅配置为websocket和禁用传输方式协商，则不需要粘性会话。
为了不改变单体的配置，还是决定在yarp中支持粘性会话。
https://learn.microsoft.com/zh-cn/aspnet/core/fundamentals/servers/yarp/session-affinity?view=aspnetcore-9.0

api部分的粘性会话需要配置为基于head
```
    "Clusters": {
      "main": {
        "Destinations": {
          "d1": {
            "Address": "http://localhost:21021/"
          }
        },
        "SessionAffinity": {
          "Enabled": true,
          "Policy": "CustomHeader",
          "AffinityKeyName": "Ak"
        }
      }
    }
```

然后改造前端请求api的逻辑，在拦截器部分尝试获取并存储Ak请求头，通常首个请求能获取这个值，
然后在后续的请求都加上这个头

这样signalR的sse或轮询方式时，应该能正常把请求路由到最初的服务器

后端不同的路由可能映射到不同的群集，所以根据httpclient的基地址来决定前端
如何缓存粘性会话id

这个与abp无关，应该定义在common中

粘性会话处理器首先应该知道有哪几个基地址，一个群集对应一个，因为后端的粘性会话id是根据群集来的
，然后请求处理时，用当前请求的url去匹配找到当前请求的哪个群集，
然后根据这个作为缓存键进行处理。
