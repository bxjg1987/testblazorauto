# 粘性会话
虽然目前不考虑前端的负载均衡，但要需要考虑进来，
后端的api上有基于signalR的实时通知，abp也默认对signalR提供了集成，意思是它不仅仅用于实时通知，还可以用于其它实时功能。

无论前端还是后端api都依赖signalR，这个框架若配置为仅websoket且关闭通信协商功能时
是不需要粘性会话支持的。

但abp默认没做这个设置，所以我们为了考虑单体和负载均衡的情况，不去动这个设置，仅仅在负载均衡场景中开启粘性会话即可。

所谓的粘性会话就是在首次请求时拿到一个cookie值，或一个请求头，在后续请求时携带这个
cookie或请求头，yarp在负载均衡时会会根据这个值把请求路由到之前的某个后端服务器上。
具体的参考官方文档

这里只记录在本项目中的设置  
默认情况下yarp是开启粘性会话的，只需要做下配置，
由于目前只考虑api部分的负载均衡，所以粘性会话策略选择header
需要改造原本的api，在前端api拦截器中，若返回中包含粘性会话id，则存储，并在后续请求设置这个值
