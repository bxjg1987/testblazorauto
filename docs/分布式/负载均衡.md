# 缓存
默认情况下的内存缓存，abp提供了实体缓存，在实体变动时会自动处理缓存，
在负载均衡场景中，A服务器更新了某个实体，若是内存缓存，B服务器中的缓存还是老数据，
所以用redis，但abp本身对缓存做了抽象，所以只需要集成redis即可。

我们的代码中只是简单使用redis，所以redis不做群集也行。

windows版redis下载地址：https://github.com/redis-windows/redis-windows/releases 
测试下载的版本是 Redis-7.4.3-Windows-x64-msys2-with-Service

Service installation:
Can achieve automatic startup on boot. Please run it as an administrator and change RedisService.exe to the actual directory where it is stored.

sc.exe create Redis binpath=C:\Software\Redis\RedisService.exe start= auto
Start service

net start Redis
Out of Service

net stop Redis
Uninstall service

sc.exe delete Redis

然后参考官方文档配置redis模块：https://aspnetboilerplate.com/Pages/Documents/Caching

# 后台作业

尽量使用abp的后台作业，不要直接使用hangfire的api

后台作业是有开关的，所以可以仅保留一台服务器执行后台作业，
但这不太好，因为可能刚好配置启用后台作业的服务器挂了，
最好是后台作业使用分布式锁，确保后台作业只有一个执行。

另一方面，若使用hangfire，貌似它可以确保只有一个线程的任务被执行。

但为了避免对hangfire的依赖，还是用abp的后台任务api，所以还是得假定整体后台作业没有hangfire的能力。

# 周期性任务worker
直接使用abp的api，注意使用分布式锁，
道理是一样，不要直接使用hangfire的api

# 通知
发布的通知会存储在数据库中，会使用后台作业发布实时通知，

支持多种通知，比如 邮件、短信、基于signalR的实时通知。

前面说了，后台作业准备用分布式锁，来确保一个任务只被一台服务器实例的单个线程执行。


# signalR
这个不仅仅用于通知，也可以用于实时数据。
在负载均衡时，api的多次调用可能被路由到不同的后台服务器实例，
但signalR长连接应该是固定连接到固定的后台服务器，
为了实现全局实时推送（包括通知），需要实现signalR背板。
https://learn.microsoft.com/zh-cn/aspnet/core/signalr/scale?view=aspnetcore-9.0
最好是在yarp中设置脸粘性会话 https://learn.microsoft.com/zh-cn/aspnet/core/fundamentals/servers/yarp/session-affinity?view=aspnetcore-9.0
再配合redis背板：https://learn.microsoft.com/zh-cn/aspnet/core/signalr/redis-backplane?view=aspnetcore-9.0

若仅考虑websoket，则不需要考虑粘性会话。

此外，若将来需要把前端部署为复杂均衡，也需要这些，因为我们使用了auto模式
