# 如果要在领域服务中触发事件，外部通常应该只是单次调用{#trigger-event-only-in-app-service}
用户的一次请求通常调用某个应用服务方法，内部可能多次调用领域服务的方法，比如一次性添加订单
及其明细时，添加明细的领域服务方法就会被调用多次，若在领域服务方法中触发事件，则一次请求会触发事件多次

还有一种情况是，一个应用服务方法中，做了某个状态的变动，之后的某个调用又做了状态变动
刚好两次变动后，状态变为了最初的状态
所以此时去发事件也是不合适的。

所以如果要在领域服务中触发事件，外部通常应该只是单次调用

# 代码放应用服务？领域服务？实体？扩展方法？
若将来需要本地化或者扩展方法需要其它服务
优先考虑参数传递过来

也可以考虑把这些逻辑放到领域服务里

其次考虑静态方式提供当前范围内的ioc容器，或scopefactory，这可以看成是上下文模式  
此方式是最不推荐的，因为这类方法粒度太小，可能频繁创建瞬时服务，而这些服务本可以在一个用例中共享
