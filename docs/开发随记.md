# 2023-12-16 blazo auto 授权
在asp.net core中 使用AuthorizationService进行权限检查，是基于策略的授权。
在abp中使用IPermissionchecker，走的是自己的授权框架，这个是因为abp是从framework进化来的。

由于blazor等新兴框架都是基于asp.net core的授权机制，所以需与abp的机制做个集成。

在服务端，最后发现自定义的AbpAuthorizationPolicyProvider可以实现集成，所以auto模式中server部分直接可用。
此时AuthorizeView AuthorizeAttribute都直接可用，因为它们内部使用AuthorizationService
在服务端部分可用同时使用IPermissionChecker和AuthorizationService进行权限判断

由于是auto，server和client共用组件，因此需要为客户端实现权限检查核心
客户端实际也是使用AuthorizationService的，

最终决定，blazor界面部分不要使用IPermissionChecker了，统一使用AuthorizationService

由于客户端和服务端权限判断差异较大，分开做

如果存在使用WebAssembly交互性的组件，请使用WebAssemblys交互性否则，如果存在使用服务器交互性的组件，请使用服务器交互性。
否则，如果已加载WebAssembly资源，请使用WebAssembly交互
否则，如果尚未缓存WebAssembly资源，请使用服务器交互。否则，在某些情况导致上述情况之一变为真 (例如，WebAssembly资源完成下
载)之前，不要使组件具有交互性。有趣的事实:实际的自动模式逻辑读起来几乎和上面一模一样
自动模式更喜欢使用现有的交互模式的原因是避免引入一个新的交互运行时，
该运行时与现有运行时不共享任何状态。例如，如果<HeadOutlet>组件通过服务器交互进行渲染，
则不希望包含<HeadContent>的Auto组件使用WebAssembly交互

# Blazor按线路保持状态
一个用户可能在多处登陆，
所以一个用户 可能有多个线路。
按用户保持状态是很容易实现的，因为我们可以通过abp session拿到用户id，从而从任何地方拿到用户状态
所以仅仅是线路状态需要特别处理，因为可能是匿名用户，不过线路可以与某个用户关联。

线路一般注册为scope，在连接和断开时做一些处理。此时可以拿到当前用户，并与线路关联

反过来，我们可以通过用户id，找到他关联的所有线路，从而做一些事情
这个是个通用功能，所以放BXJG.Utils.RCL中

# 通知
>请先熟悉abp的通知系统\
注：发送实体通知时，必须订阅才会发送

由于我们的系统使用blazor server，所以自定义实时通知器

需要借助事件总线与UI交互，由于abp8.4-的版本，事件总线有线程安全问题，运行时注册事件可能引起报错，所以目前我们使用自己的Zhongjie

由于abp的通知是发送给用户的，理想的情况是每个用户一个Zhongjie实例，这样给指定用户发送通知时，使用此对象触发事件，从而界面弹出通知提示。
这样需要定义一个全局容器，

应用和领域层发送通知，UI层实现实时通知器，由于是个通用的玩意，定义在BXJG.RCL中即可。

## 用于界面通知的Zhongjie
BXJG.Common中注册了一个全局的Zhongjie，它用于整个应用层序，由于abp自身的事件目前仅仅是有bug，9.x版本应该会修复此问题，所以我们的程序中
最好别用全局Zhongjie，因为将来可能换回abp的事件方式。

不是所有事件都需要发送通知，比如某些事件的产生，仅仅是界面更新就行，无需特殊的还去通知用户，因为通知是要存取数据库的

目前仅仅打算将Zhongjie用于界面交互，比如 组件与组件之间的交互，或业务层通知界面。

可以使用全局Zhongjie来做界面通知，和组件通信，因为它可以用Key的层次结构来限定范围，不过有点担心大量key会不会有性能问题
但即便是每个用户一个zhongjie实例，也都是存储在内存中的。

## 决定
Blazor按线路保持状态 小节说了，按线路关联Zhongjie，线路可以关联或不关联用户，反过来可以通过用户拿到所有关联的线路。
所以这样比较完美，abp的事件总线bug修复后再改下，或不改，因为它是全局唯一的。
