# 场景
比如学校系统中有系统管理端、学校管理端、家长端和学校端。
每个端对应一个应用层，它们可以使用统一的host层，也可以单独定义自己的host层。  

**所有端的用户都是单独定义的，但它们都继承User类。  
没有使用早期的各用户通过关联关系关联到User表**


各应用共享底层的ZLJ.Core和公共应用服务ZLJ.Application.Common。

各应用也引用UI公共库，如：ZLJ.RCL


另一种情况是干脆不考虑在单进程中承载多应用，而是完全独立托管每个应用。这种方式在一些小项目中太麻烦了。
后期维护、部署都麻烦。

# 如何使用
目前已实现，在IAbpSession中提供了GetAppKey方法，前端在cookie、请求头或header里提供appKey

# 共享部分可能需要识别是哪个应用
公共应用服务、底层的Core以及UI共享库中的某些逻辑可能需要识别当前应用做不同的处理，比如小程序登录中，
使用了统一的小程序登录处理类，不同的应用类型可能有的需要自动注册，有的在没有找到绑定用户时反馈登录失败。

总之某些场景需要识别当前应用。

# 使用租户识别类似的逻辑识别应用
首先说不能仅依赖用户的Cliam识别应用，还有可能同一个用户进入不同应用的场景。
有时用户还没登录，但需要识别当前应用。

可以不像abp识别租户那么复杂，硬编码实现：从请求头、querystring、cookie中获取。

# 前端提供应用标识
后端只能识别请求，无法识别到底是哪个应用，尤其是用户未登录时，极端情况同一个用户在多个设备上登录到不同的应用。
所以区分应用的起点是前端。

在pc端可以在登录时就区分应用了，进入各应用后续请求往往是请求不同的接口，前后端都可以硬编码应用标识。

特殊的是在微信小程序中，共享同一个小程序appId时，
建议的方式是持久化存储当前应用id，在公共的http请求中从持久化或缓存中获取，做不同登录时先处理持久化。

或为登陆前手动提供应用标识，登陆后持久化存储，后续请求从持久化存储中提供。

另一种思路是不吃就好，在关键页面内存中定义appKey全局变量


# 获取当前应用挂在到session上
对session的扩展应定义在Core中，或不考虑core访问应用，就定义在ZLJ.Application.Common中

# 全局应用容器
AppInfo 应该设计得轻量，应定义在ZLJ.Core.Share中，容器应定义在ZLJ.Application.Common中 
这个前期可以不考虑，以后需要再实现

大体思路是在ZLJ.Application.Common.Share定义AppInfo，在各应用模块中注册自己的应用信息
