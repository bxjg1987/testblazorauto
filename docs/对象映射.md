请先参考abp官方文档对象映射部分

# 在客户端部分注册IObjectMapper
abp中，AutoMapperObjectMapper是IObjectMapper的默认实现。
服务端部分无需注册此接口，因为abp已经这样做了。
要集成的第一个步骤是前端要在ioc注册，并使用abp的默认实现，由于这是通用的，所以在ZLJ.RCL的依赖注入的客户端部分
ZLJ.RCL/Extensions/DIExt.AddZLJBlazorClient
```
services.TryAddSingleton<IObjectMapper, AutoMapperObjectMapper>();
```
# 在客户端部分注册automapper本身
上面说的IObjectMapper接口是abp在automapper基础上抽象出的接口，具体实现是automapper，automapper本身还需要注册和配置
参考：https://docs.automapper.org/en/stable/Dependency-injection.html
这个注册不能是共享的，仅仅是在客户端的启动时注册，由于需要注册时指定包含profile的程序集，所以最终在ZLJ.Admin.CoreRCL的Program中注册
```
builder.Services.AddAutoMapper(typeof(Program),typeof(ZLJ.RCL.AppContainer));
```
我们指定了了当前程序集和RCL程序集，所以公共的映射在RCL中定义automapper的profile，具体应用则在具体应用的CoreRCL项目中定义automapper的profile

# 在服务端UI部分关联profile
前面一个步骤为automapper指定了包含配置profile的程序集，但默认情况下服务端部分不会自动包含这些程序集
所以在服务端处，按abp标准的方式注册，在ZLJ.Web.HostBlazor的Module类中
```
Configuration.Modules.AbpAutoMapper().Configurators
                .Add(cfg => cfg.AddMaps(typeof(ZLJ.RCL.AppContainer),typeof(ZLJ.Admin.CoreRCL._Imports)));
```

# 开发
这样在UI开发中，需要映射时注入IObjectMapper使用即可，也可以按abp文档对象映射部分的说明，直接使用automapper的api，即：IMapper接口，但我们不推荐。
此外，公共部分的映射配置，在ZLJ.RCL中定义automapper的profile，具体项目则在具体项目的client部分定义映射配置文件。

# 其它说明
在抽象详情编辑组件中，我们默认将dto映射为editdto，这个映射配置虽然可以，但不推荐在接口层中去定义，因为对象映射属于具体实现部分，而非规范。
