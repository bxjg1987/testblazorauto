# 继承.net自己的配置和abp setting的应用程序级别的配置
abp有自己的setting，分应用、租户和用户级别。

.net core 有自己的配置系统。

某些非abp模块会直接使用.net core的配置系统（包括options模式），
希望这部分还是使用.net 自己的配置模式。仅获取配置。
注意，它仅关注应用级别的配置，目前不考虑租户和用户。

让更新使用abp的setting模块，

.net core的配置有个功能是配置源更新后，获取配置可以拿到最新值。


实现思路
自定义配置源和提供器
配置提供器在启动时，直接从数据库查询，因为启动时还没有服务注册，没法用ISettingManager或存储库
并注册全局事件总线。变化时通知外部。

后续abp事件监控设置实体修改事件，触发事件，可以用Zhongjie.Instance

或者参考这个：https://makolyte.com/event-driven-dotnet-how-to-use-query-notifications-in-sql-server-to-monitor-database-changes/
监控sqlserver 得到变更通知。上面的方式不需要数据库特殊配置，也不需要关心具体数据库


注意：IConfiguration内部引用ConfigurationProvider的，所以提供器的值更新，配置应该也可以拿到更新，只是没通知应用程序。

.net core 自定义提供器的通知机制

# 另一种更简单的方式
按约定，在开发中，应用配置通通使用.net core自己的方式，不要使用的abp的方式。
只有租户或用户相关配置才使用abp的方式，这样就不需要做任何更改和适配

# 前后端配置同步
前端server时，通过scope隔离配置
wasm时，单例。

前端更是需要.net原生配置方式。若按上面的简单配置约定，应用级别的配置压根没使用abp的方式，那么前端应用级别的配置可以放心使用.net的方式。
但是如何同步配置呢？
一种方式是把api和前端需要的配置放独立的json文件中，双发都add进来。
或者干脆两边独立复制一份。
后端和前端部署在一起时有用。

另一种方式是，前端调用api从后端获取。前端启动时获取并配置。由于仅关注应用级别的配置，这也不是不行。
这样前端仅保留后端api地址的配置节点
前后端独立部署时有用，但需要处理配置更新的问题。可以考虑通过定时任务或signalR主动推送
