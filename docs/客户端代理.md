
我们的接口被分组为：  
多个应用 + appcommon + utlsapp  
一般它们承载在同一个Host中，我们使用一个统一的httpclient（名为BXJGHttpClientExt.HttpClientName），
它注册时设置基地址，然后在创建httpclient实例时，上面的 多个应用 + appcommon + utlsapp  都在基地址上加上自己的地址前缀。
如：https://xxx.xx.com/api/admin/

utils app clientproxy 提供扩展方法用于注册基础httpclient，并配置assesstoken和abp ajax解包 拦截器。

Application.Common.ClientProxy 是统一的客户端服务注册中心，它内部会调用上面的 utils app clientproxy 的httpclient注册

每个应用的前端都是单独的项目，它们的客户端注册中就包含Application.Common.ClientProxy的服务注册。


上面说了注册部分，下面说说如何使用httpclient

每个应用提供了自己的IHttpClientFactory扩展方法用于创建hc实例，所以使用它 createAdminClient  createUtilsClient  createCommonClient 即可。

utls.rcl zljrcl 是通用的rcl库，多个应用共享，在定义抽象组件时，它不晓得应该用哪个具体的创建方式，所以提供一个默认的，且允许修改。
比如utls.rcl中默认的就是创建utils的httpclient，而zlj.rcl中则覆盖它，改为创建common的httpclient。
而admin rcl 则覆盖了utls.rcl中的默认值，改为创建admin的httpclient。

## 解包ajax中的异常处理
前端的异常处理往往只需要提示用户。请求后端接口 响应 解包是在utls中的messagehandler拦截器实现的。
这个库没有引用具体ui，而错误提示往往是具体ui库提供的。
需要提供一种机制。

由于注册httpclient时在前端应用启动阶段，而此时界面还没有，所以此时注册消息服务不太合理。

另一个办法是httpclient的创建是通过工厂扩展方法来的，在这个httpclient创建的扩展方法动手脚。

最后可以定义一个接口，在解包时调用这个接口，来弹窗显示错误消息。目前使用此方式。

还可以考虑事件总线。

或不处理，只抛出异常，但这种情况批量操作就不好处理的。

可以使用一些开源库根据swagger自动生成客户端代理

我们使用的是硬编码+泛型的方式。

首先定义一个类，它对HttpClient进行扩展，结合了abp接口的特点

Post
Get
普通数据的CRUD和Provider
树形数据的CRUD和Provider

然后通过统一名称来代表整个项目的HttpClient，可以理解为默认情况下一个进程就一个HttpClient类型，注意是类型不是实例。
HttpClient的名称一样，实例则是每次使用时创建，实例的释放由工厂负责，所以不需要释放。

由于不同应用的api的基础地址不同，通过扩展IHttpClientFactory来创建utils、common和admin 不同应用需要的httpclient实例，它们的名称相同，但是api基础地址不同。


之后在普通数据和树形数据的抽象curd proivder组件中，直接通过IHttpClientFactory来创建HTTP client实例
由于抽象组件中需要引用httpclient，所以将获取httpclient实例定义为抽象树形或虚属性。

由于admin中所有具体业务组件是直接继承ZLJ.RCL，不希望在admin又定义一遍抽象组件，以免继承层次太深，
就导致admin中所有的组件都需要重写获取HttpClient的属性，并提供admin对应的httpclient创建方法。
所以改下设计思路。

抽象组件中定义的HttpClient属性默认从 一个静态委托获取，而不是直接获取，
但这个属性还是保留为虚属性，以便子类确实需要重写。
然后在admin项目配置时，指定这个委托，这样默认就变成这个委托了。
