webapi中controller接受用户请求，controller做简单的数据校验，然后调用应用服务，待应用服务返回后，controller响应给用户，所以核心是应用服务。  

什么是用例？  
我的理解是：做某件事，如：用户输入账号密码点击登录，这个“登录”就是一个用例；再比如电商系统中的“下订单”也是一个用例；最常见的curd每个操作都是一个用例。  
一个用例涉及到多个内部操作，比如登陆时根据登录名去数据库查找用户 -> 判断密码是否匹配 -> 判断用户是否被锁定 -> 记录登录日志 ...，这些所有的步骤只是为了做登录这件事。对于前端来说，登录就是一个用例，一般对应到一个接口，也就对应到后端的一个应用服务中的一个方法。

多个相关性较高的用例对应的接口方法通常定义在同一个应用服务类中，如：登录、找回密码、修改密码、等都是与账户相关的，所以都定义在AccountAppService中。

我们不打算使用DDD，所以应用服务就是编写核心业务逻辑的地方。

# 去掉抽象应用服务中的构造函数
abp中为了简化curd，提供抽象的crud应用服务，包括同步和异步版本。
你继承它时必须提供一个构造函数，因为抽象类中定义了一个长长的构造函数，
虽然ide提供了快捷方式让你快速实现此构造函数，但看着还是不爽。
我们可以定义一个类，继承它，然后在内部通过反模式来去掉这个构造函数。

## 防止内存泄漏
由于内部使用反模式，所以需要ioc容器，无论是注入IocManager还是调用它的静态单例对象，都是全局容器。
若在这个全局容易解析一个实现了IDisposable的服务，若恰巧此服务是范围或瞬时注册的，那会造成内存泄漏。

经过测试，属性注入IServiceProvider是安全的。
