# 使用方式

1. 在模块pre方法中，IModuleConfigurations.BXJGUtils().SelectableTagProviders.Add(...
   这样可以定义某个实体类的某个属性要使用tag的可选列表，最简单就是用内存硬编码形式。  
   也可以从其它源获取，如：settings、通用字典、  
   若希望用户自定义的tag也参与到可选列表，则在同类型key下的委托列表增加一个委托，从tag表取获取数据（参考BXJGUtilsModuleConfig.GetSelectableList）
   注意返回orderIndex影响最终返回tag列表的顺序，数字越大就越会排在前面。
2. 前端调用utils/TagProvider接口
3. 提交保存数据时，使用TagManager.Set
4. 将来查询实体的tag时，由于需要跟随实体的权限，用实体自己的应用服务做查询，并且IQueryable<Tag>已经提供了扩展方法





   可选tag

   

   这个用通用字典就行了，当添加自定义tag时自动增加。

   默认可以按租户隔离。

   

   此外一个项目可能存在多种tag类型，字典建多个就行，前端通过name区分。

   

   

   tag存储

   

   方式1：直接存储为以逗号或空格分割的字符串 （简单情况推荐）

   

   方式2：json字符串

   

   后续tag处理无非是搜索或统计，json和动态属性都算是结构化的，数据库本身就支持json操作。第一种仅仅支持like模糊查询。

   考虑到兼容性，json底层可以这样存储，但是上层不应该使用json查询api

   

   方式3：动态属性

   这种情况根本不行或者说不合适，因为动态实体属性是与实体类型相关的，

   若非要这么做，原本abp提供的相关接口也会被污染。

   

   方式4：utils种设计一个tag模块

   

   # 可选tag

   ## 应用服务

   最好是提供一个默认的应用服务，它根据类型名和可选的属性名获取可选tag列表。
   默认权限登录用户即可，或由数据源决定。

   数据源下面分析

   ## 内存（委托）

   这个不好实现租户隔离，也不适合运行时增加，但它最通用，因为后续的每种方式都可以看成是一种实现。

   在模块配置中存储委托列表，列表中的每个委托都是一种获取方式。
   为了满足权限控制，要给每种方式指定权限方式

   多种方式合并去重的结果作为最终返回值

   

   ## 通用树字典

   但是tag本身有name和displayName，而字典的name是标识类型的，不能用。
   另外树是实体，tag最好是按用户、租户隔离

   

   ## 直接从tag表中获取

   另一种方式是在已有的数据中获取，这样不需要单独维护一个可选的列表

   这种查询性能略低，也不容易配合缓存，因为实体数据可能变化太频繁了，需要每次变动时去比对后决定是否让缓存失效
   不过页没区别，就算单独用字典，用缓存的话也有一样的问题。

   灵活度低，将来加新能力时比较费劲

   ## settings

   适合应用、租户、升至是用户隔离
