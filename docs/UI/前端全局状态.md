# 当前的实现
此文档仅描述此项目前端状态管理，blazor本身的状态管理[参考](https://learn.microsoft.com/zh-cn/aspnet/core/blazor/state-management?view=aspnetcore-8.0&pivots=server)

我们的项目使用blazor web app auto模式独立部署 + 后端api的方式。所以属于前后端分离模式。
刷新页面时 首先执行普通的http请求，之后再进入server或wasm模式，经过测试，一个服务注册为scope，在http中获取和server中获取是不一样的，前者的范围是http请求，后者是长连接电路。

abp默认提供getall(AbpUserConfiguration/getall)和getcurrentlogin(api/services/common/Session/GetCurrentLoginInformations)接口
getall
包信息如：所有本地化文本、整个项目的权限字符串（不是当前用户以授权）等等...
它在登录前可以调用获取一些全局信息，登陆后可以再调用一次，此时会包含如：当前用户已拥有的权限等信息。
有哪些配置可以参考：AbpUserConfigurationDto
**后端的设置目前仅考虑setting，而前端的设置是个缝合怪，里面包含：菜单、权限、特征、设置等等。**

getcurrentlogin...
这个是仅登录后调用，获取当前用户信息相关

所以getall某些信息是包含getcurrentlogin的，为了与abp官方保持一致，这里也按这种使用。


上面俩接口说的登录信息，权限、菜单、设置 等信息就是状态。
我们希望的是，无论在server还是webassembly模式中，组件访问状态的方式保持一致。

结合blazor本身的状态管理和我们的开发模式以及这里的场景，做如下设计：
直接用ioc作为状态容器，getall和getcurrentlogin定义为task<dto>类型，以scope方式注册到ioc容器中，这样无论是server还是wasm或http请求时，
需要用到状态时自己取await，比如授权处理器检测权限时。
scope注册的，server模式是是与电路关联的范围，wasm模式是单例，刚好是我们希望达到的效果。

http请求，通过app组件传入参数到router组件，由于要在http请求到server或wasm转换，还得维持状态，所以涉及到序列化和反序列化，这是同步的。
另外通过PersistentComponentState来从server到wasm进行传参也是一个道理，涉及到序列化和反序列化，且是同步的。
虽然请求后端api也有序列化和反序列化，但上面两种方式是同步的，界面无法快速显示出来，而server和wasm各自请求后端api是异步的，界面可以快速显示。

## 状态同步
前端开发的组件中，不应该复制全局状态，而应该直接引用它，这样状态变化时，才会立即生效。

后端状态变化时如何同步到前端，按上述设计，用户使用系统期间，所有状态都不变，即使后端api有变动。
理想的情况是，后端状态变动，按“实时”文档说明，推送通知给前端，前端刷新状态。
由于多个其他用户可能在短时间更新多个状态，比如管理员调整了菜单，管理员B，调整了设置，管理员C调整了权限,
所以前端得到通知是，要用延迟覆盖组件 加载，

是否需要单独更新
AbpUserConfigurationDto的每个属性对应一种状态数据，是否需要分开同步？
abp默认只提供了getall，如果不想做太多定制的话，就用这个就行了，比较配合上面说的延迟覆盖执行问题也不大，这样更简单。

抽来讲，api端状态变更只需要发送通知给前端，可以直接使用INotifyPublisher，但它默认实现发送时，若在线用户少就直接发送，在线用户多就用任务发送
另外它会持久化消息。

对于我们这里同步状态的场景，仅需要给所有在线用户发送，且不需要持久化，所以不能用通知。
后端apiabp提供了IOnlineClientManager，所以不需要我们使用hubcontext去发送实时消息给客户端。

## server模式到wasm模式的状态传递
之前分析过，不用将大量数据通过级联参数或PersistentComponentState在服务端和客户端之间传递数据
因为会有序列化和反序列化，还是同步的，界面会卡。
还是让界面快速显示，状态重新统一从服务端拿。

# 全局状态
我们是用的blazor web app auto模式 + abp后端api
用户访问系统时，从后端一次性加载与当前用户有关的全局状态，在用户使用的这段时间，这些状态不变化，只读。

>blazor官方文档有状态管理，建议先阅读。但跟我们的设计不同。

我们的思路是，定义一个全局状态对象，里面包含：当前登录用户的名称、所属租户、特征列表、设置列表、权限列表、本地化列表等等...
server模式时，每个blazor长连接（电路）一个全局状态对象，它在服务端内存中。
wasm模式时，直接单例即可，它在浏览器的内存中。
http请求期间，仅初始化部分数据，看此文档后续。

## 为啥不直接用ioc作为状态容器？
状态数据有上述特征，跟当前用户（如果有）相关，短时间不变化，把这类数据看成一些列数据，
把这一系列数据定义在一个全局状态类中，好查看，访问，管理

否则从ioc拿对象，需要记住那些状态对象的名称，
不过这种也可以在各抽象类中定义强类型的引用

不过本质上都是全局对象，任选一种方式即可，我们目前用的是定义全局对象的方式。

## ioc注意
全局对象以scope注册到ioc，客户端会自动形成单例，主要是server端需要注意

首次访问或刷新页面时，此时从http请求中获取的全局状态对象是与此请求关联的，
而后底层长连接（电路）建立，进入blazor server模式，此时再获取全局状态对象，是跟此长连接关联的，也就是一个长连接一个scope
blazor server后续其它组件获取scope的服务，都是同一个。

**关键问题在于http请求的scope和长连接（电路）的scope不是同一个。**

## 延长服务端http请求中的全局状态存活时间
wasm模式时，即便是以scope注入到容器的全局状态，也是单例的，所以在用户使用系统期间状态一直存活。
server模式时，由于状态在电路关联的scope中，所以在用户使用系统期间也没有问题。
重点是，无论哪种渲染模式，在首次加载或后续刷新页面时，是先已普通http方式请求页面，后续再自动决定进入静态、server还是wasm模式。
可能某些场景，需要在http请求期间访问全局状态，典型的就是授权检查时，需要访问当前用户已授权的权限字符串。
所以严格的来说，全局状态会 存在与 blazor wasm、blazor server和http请求期间。

既然是全局状态，不经常变化，短时间内，多次请求每次都从后端api拉去全局状态，太浪费，
所以需要通过session，来让http请求中的全局状态，存活的时间变长。

从app传参到router组件是可行的，但测试报错，好像是大小限制，因为涉及到序列化，但也不完全确定。
本来这种方式可以在http请求时初始化全局状态，然后通过app传递给router，而组件参数会自动在server和wasm同步，
这样只需要初始化一次的。
奈何目前测试不通。\
**另外这种序列化和反序列化都是组件参数，应该会影响到UI渲染，而且不同场景自己通过ajax请求后端api，则是异步的，不会影响UI渲染。所以大量状态不要通过组件参数方式来做**

## 状态初始化
http请求中初始化一次
http请求期间和进入blazor后都需要权限检查，但其它状态在http请求期间好像用不着。
所以在http请求时，仅初始化下权限字符串，并通过上面说得方式session延长 权限字符串的存活时间。

blazor中初始化一次
请求后，blazor自动决定进入何种渲染模式，它们始终会执行路由，可以在路由中初始化整个全局状态。

**权限字符串比较特殊，它在http请求期间，和进入blazor后续都需要用到，它应该作为全局状态存储，而不是特殊处理**

权限检查处理器统一从全局状态容器获取已授权的全局字符串列表，详细的参考 授权文档。

# 除了一些全局的数据外，提供下拉框的数据如何设计？
比如选择客户的功能
整个项目中多个地方需要选择客户，后端api已经做了高度抽象
前端定义了泛型的select组件，它内部会调用接口填充下拉框数据。

但假设有个表格，它直接提供编辑功能，它的某列用来选择客户，
此时若列中每行都提供一个客户选择的select，它们各自提供可选列表，会发送很多http请求后端api
最直接的办法时在当前组件中，仅查询一次接口，获得的数据赋值给所有 客户选择下拉框。

但若有多个组件都使用了此方式，每个使用的地方都要自己去准备下客户列表数据，
另外系统中除了客户，还有很多需要提供下拉框数据的情形，都按此设计编码量比较大。

可以通过级联值的方式，在表格外层使用级联值方式，这样所有使用的地方都简单了
进一步的方式是定义一个组件，它内部包含这个级联值提供的逻辑。

更进一步，提供一个泛型版本，内部对于后端的数据provider接口上，这样一次性解决所有下拉数据的获取

>通过ioc提供级联值
>https://learn.microsoft.com/zh-cn/aspnet/core/blazor/components/cascading-values-and-parameters?view=aspnetcore-8.0#root-level-cascading-values
>里面提供值的委托不是异步的，若级联一个task，担心序列化的问题
