此文档仅描述前端部分的授权处理，后端api是abp默认方式。

基于角色和功能码是常见的需求，abp做的后端api已经实现了。

前端登录后，会拿到全局状态，其中包含已授权的功能码列表，前端授权检查时，检查目标功能码是否在列表中，若在，则授权成功。

前端的授权主要是用AuthorizeAttribute标注页面，控制页面是否可以打开
通过IAuthorztionService检查指定按钮是否有权限，若有则现实，否则隐藏
所以真正的权限控制还是后端api控制的，前端仅控制现实。

>阅读此文档前，先看 全局状态.md

授权页面两次检查

按asp.net core 默认授权方式是在启动时，全局注册所有授权策略，按此方式需要在初始化时从后端api拿到当前登录用户的所有已授权的操作码列表
然后遍历注册授权策略，并且将策略名称存储在授权依据中，因为授权检查处理器是检查的依据，而不是策略本身。

这种方式比较浪费，所以自定义授权策略提供器来处理，
它先看有没有全局策略与当前检查的授权策略名称匹配，若没有就把当前请求的授权策略对应到操作码，创建策略并返回。

授权处理器通过全局状态中获取已授权的操作码列表，进行比对。
