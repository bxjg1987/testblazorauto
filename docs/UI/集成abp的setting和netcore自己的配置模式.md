# 简单设计

这里是简化的设计，文档后续有之前考虑的标准设计。

由于是前端server wasm公用组件，所有状态的访问都可以通过AbpUserConfigurationDto直接访问，不使用 配置模式和abp的settings
由状态管理文档中的，实时同步 确保一致获取新数据。
这么看，只有后端，非abp库才需要 setting与配置的结合，而前端 既不使用配置，也不使用setting，而是整体从后端获取AbpUserConfigurationDto，然后直接访问
菜单、特征、本地化、setting等数据。。

若某个库，在api和前端公用，但这个库本身需要配置呢？
常见的就是 某个实体 根据不同状态，决定可以做不同操作，这种逻辑前后端都要做。
前端用来决定按钮是否显示、是否禁用。后端使用这种逻辑来实现业务功能。这些逻辑可能需要用到配置。
这种情况应在后端做，然后将结果反馈给前端，而不是共享库，主要是考虑不用blazor的场景。

当然，若仅考虑blazor前端，共享库还是非常有用的。
简单起见，目前不考虑这种情况


# 标准设计


这个版本的abp出生比较老，那是netcore没发布，所以出现的情况是
abp有自己的一套settings功能，而.net core又使用的configuration模式

>options模式在建立在configuration模式上的，所以我们只有设计到configuration模式就行了，options模式保持netcore自己的方式即可。

由于前端是独立项目，不属于abp模块方式，它引用abp也仅仅是使用里面的一些简单类，如：dto
希望前端保持netcore的配置模式

>netcore的配置模式仅关注获取配置，以及在配置被更新后得到通知，但并不关心配置本身的修改。

另外abp settings分应用、租户和用户级别的配置，但针对前端来说，无论是server还是wasm，登录后获取到的配置应该直接就是合并后的，后面的优先级更高。

设计如下：
某些在启动阶段必须的配置，如：后端api地址 这个直接定义在appsettings中
其它配置都异步从 全局状态获取，它是task，需要await，这样界面可以及时渲染，配置稍后更新。
具体来说就是自定义配置提供程序，首先获取空配置，然后开启任务去加载配置，获取后通知
后续可以更进一步，后端配置更新后，可以通过  “实时文档”部分的说明，通知到前端，前端重新加载后通知iconfiguration
