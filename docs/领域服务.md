# 前言
因为DDD复杂性，所以本项目并不采用。但领域服务还是有一些作用的，下面说说本项目中领域服务的思路。
它是说明，它与应用服务层、外部系统、仓储之间是什么关系（这几个概念单独文档说明）。


# 多个应用的公有逻辑定义在领域服务中。
比如在电商系统中，初略的划分为 前端和后端两个应用，它们有各自的应用层，但它们存在一些共用的功能。

# 功能范围控制
就是领域服务中一个方法定义的功能别太多，不同的应用中某个方法可能需要调用多个领域服务方法来组织逻辑，
若领域方法中一次性定义太多功能，这个方法就不够通用了。
也不能定义得太细，太细工作量大，也木有意义，因为最细就是一行代码，这个时候领域方法就没意义了。

功能范围具体定义多大自己根据业务去划分，也可以先粗一点，在根据开发进度慢慢改细

在细粒度的领域方法基础上，可以提供一些包装方法，它里面实际上是调用多个细粒度的领域服务方法。
这种方法可能在A、B应用中通用但在C应用不适用

# 外部服务与仓储
某种意义上来说一个外部服务方法的调用或一个仓储的方法，也可以看作是我们上面描述的领域方法。
按上面说的领域包装方法，领域方法是可以调用外部服务或仓储的。

>之前一直纠结一个问题，在新增一个数据时若逻辑比较复杂，就定义在领域方法中，但是最后是否应该在这个领域方法中调用Insert
由于abp提供的事务，最终决定可以调用，做后续逻辑需要放弃insert，抛出异常阻止即可。
