# 简单
尽可能设计简单，因此需要规划代码生成器的范围，如：仅考虑本项目、仅考虑代码生成而不是文本生成、使用单一模板引擎等。

# 与AI的关系
AI是大脑，它按你的任何要求生成任意代码，更灵活，但不适合批量的按固定规则生成代码。
我们这里要做的代码生成器，主要是根据模型+模板生成固定的代码，更简单高效，但缺乏灵活性。手动挡。

# 考虑适用于此框架的项目和其它类型的项目
由于限定在代码生成这个场景中，所以此代码生成器除了适用于此框架的项目，也考虑其它类型的项目。

# 独立项目，不使用abp
单独一个项目，因为要设计简单，也考虑用于其它项目，所以不要使用abp，它仅仅是一个工具

# 项目定义
一来是项目本身存在一些数据定义
其次项目作为模型和模板的容器。

包含的字段：项目名称、英文名(也可以自动生成全拼音)、

# 模型定义
就是实体定义，或表定义，比如：客户信息
包含：模型名称、数据表名称（为空时根据约定生成）字段定义、等等
字段定义包含：字段名称、数据类型、是否可空、默认值、数据长度、时间格式、是否可以排序、列顺序等等...

一个模型只能属于一个项目，关系更简单，事情更简单。

# 模板
之前考虑同时支持js和razorlight，这样设计上会比较灵活复杂
现在只考虑一种模板引擎，razorlight，这样更简单

>由于blazor8已经支持，不过他貌似不支持在线修改模板，改了需要编译
参考：https://learn.microsoft.com/zh-cn/aspnet/core/blazor/components/render-components-outside-of-aspnetcore?view=aspnetcore-8.0

一个模板只能属于一个项目，即使多个项目都使用此框架，不同项目的模板也分开定义，复制也许，这样更粗暴，简单。

# 模型定义与模板关系
多对多
定义模型和模板的关系方便后续选择，选择模板后，自动列出可选的模型，或者反过来。

这个关系中定义模型时通常不用指定模板，而更应该在定义模板时指定关联的一个模型，因为模板依赖模型，而模型不依赖模板。
由于使用的razorlight，一个模板只允许关联一个主模型，若一个模板要引用多个模型，则要么定义多个模板，要么定义一个包装模型。

# 后续脚本
除了代码生成，还需要一些后续脚本的执行。
若仅考虑此框架的项目，其实可以硬编码，但考虑到非此框架的项目，变数就比较大。
所以考虑使用脚本的方式。

脚本动态加载并执行。
也可以考虑同时支持按接口和实现+反射的方式执行
所以一个脚本通常是一个dll或一段代码文本脚本。也可以在一个dll中定义多个脚本实现类。
动态脚本其实也跟dll一样，实现固定接口，只不过实现部分是加载并执行脚本。

>针对与此框架的项目，因为场景固定，直接将常用脚本定义在单个dll中。

一个脚本一个线程，做一些列事情。多个脚本并行执行。

脚本也只属于某个项目

**参考：综合分析**

# 综合分析
这么看，主生成流程、后续脚本、事件，其实是一回事。最灵活的情况就是整个生成过程全部调用插件，一个插件一个线程，多个插件并行执行。
主项目执行定义这些核心概念，以及它们的关系。

在此基础上默认实现一个主流程，在主流程中回调事件。这样框架就变得及其灵活了。后续脚本也就变成事件了。
大部分的自定义需求只需要定义回调，高级的情况是替换或增加主流程。

主流程的默认实现，在执行期间传递一个上下文，仅包含主流程执行过程中的信息，不包含全局配置信息

回到我们的原则，尽可能简单，所以决定主流程定死，仅考虑基于事件的扩展，且针对于此框架提供默认的事件处理器。
