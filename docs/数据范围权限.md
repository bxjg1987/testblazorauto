\## 早期数据准备

uow是在应用服务aop准备的，可以自定义拦截器，由于是动态拦截器，很容易获取uow，由于数据是最终执行查询时被使用，所以早晚注册拦截器应该是无所谓的，

此时需要一个问题，应该在这里知道元素据，

要么不管，数据多就多点，直接传递给dbcontext，它内部根据目标实体类型去过滤，由于缓存+快速执行，会临时占用下内存

或者在应用服务方法上加attribute，aop拿到它上面的元数据name，这样数据量更小。

attribute应用在类上，由数据过滤器开关配合

好像有点麻烦，业务attribute不确定内部查询会用到多少实体类型，还是应该传到下游



早期拦截的数据处理后才传到dbconext，这样可以降低内存，最后传递下的数据格式为：

\[

&nbsp;     { 数据类型1 ,DataPermissionGrantType, \[ oucode1,oucode2... ],\[ouid1,ouid2] },    

&nbsp;   { 数据类型2 ,DataPermissionGrantType, \[ oucode1,oucode2... ],\[ouid1,ouid2] },

]

数据类型已经去重，DataPermissionGrantType需要ou时才有后面的数据，code直接做or likie，ouid做in



可读性更好的化，就直接传dbcontext中统一处理，但早期做处理后，还可以做缓存

缓存

使用租户缓存，一个用户的角色、组织单位列表，基本不变，所以可以直接按用户 缓存数据权限列表



最终在uow.Items上存储GrantType和DataOrganizationUnitid，列表，这样比较简单，

也免得目标数据都去join组织单位表。

但若组织单位列表都，in查询估计有点慢
