
# ZLJ.Web.HostBlazor项目
它跟ZLJ.Web.Host是平级的，一样引用应用层，前者提供webapi，ZLJ.Web.Host提供blazor auto模式的ui
在此基础上还实现了server和assembly模式的共存，从而实现所谓的渐进式分离。

# 当前项目ZLJ.Web.BlazorAuto
它使用标准的blazorauto模式，无论组件是运行在服务端还是客户端，它总是通过webapi与后端交互，
它是一个妥妥的纯前端，只不过它使用auto模式。

# 此模式的登录和身份验证流程
用户首次请求受保护的页面
blazor组件通过服务端的AuthenticationStateProvider获取用户状态
获取失败跳转到登录页
输入账号密码通过webapi登录拿token，由于是纯前端，拿到token就算结束了
将token存储在服务端的AuthenticationStateProvider
此时切换为客户端模式，通过持久化状态机制，客户端尝试从持久化状态获取身份验证状态和token即可
这样，首次不会出现在server时登录了，再client时还需要登录的问题。


若一开始客户端就已经是就绪状态的呢？
无论如何首次都是请求静态页面，此时基于cookie方式获取是最自然的，再通过持久化状态方式传递到客户端。

因为无论如何，都需要一种持久化的方式，由于auto的最外页是静态的，所以用cookie是最只管的持久化方式。
