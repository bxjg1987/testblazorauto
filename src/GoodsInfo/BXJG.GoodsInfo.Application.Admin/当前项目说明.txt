

# 场景
客户是做复印机租赁的，需要进销存管理功能。
客户那里的物品主要有：整机、耗材、配件、办公用品、还有些公司内部使用的物品，比如维修工具
这些物品类型不同，属性不同
某些物品可能只做采购和库存，而不做销售，比如公司内部使用的维修工具等。
将来可能增加新的物品类型，或新业务。

# 聚焦物品
我们希望设计细粒度的模块，方便在其它项目中渐进式的使用我们的模块
所谓的渐进式是在另外的现有项目中，只引用我们一个很小的模块，且这个模块足够小，没有太多依赖其它模块。
我们可以把物品管理单独定义为一个模块，因为它是核心的，比较独立的功能，可能另外一个系统只做库存管理的，它可以使用此模块来完成
物品管理，而自己开发一个库存管理功能。

# 原子模块
按理说物品管理是最小粒度的模块，它应该属于基础数据模块的子模块，但是为什么没有定义一个基础数据模块呢，
因为不同的系统可能需要不同的细粒度的模块，如果在基础数据模块中定义了很多原子模块，那么也许我只想用其中一个小模块
但我得被迫引入整个基础数据模块。所以将物品管理单独定义为一个原子模块。

# 子类相关服务类型注册问题
不同物品类型都有一系列相关类型，Manager、Repository、AppService等
一种办法是在模块调用方注册自己的物品类型GoodsInfoTypeDefine时指定相关的服务类Type
另一种办法是定义相应的泛型抽象，如：Manager<TRepository> AppService<TRepository,TManager>
一个管理类型可能被多个地方使用，这种场景使用方式1更灵活，一次注册多次引用，相反，每一个依赖此类型都得定义一次泛型
另一个好处是模块调用方只需要在模块启动阶段通过一注册一次。
但对于具体物品类型的管理子类，如：ZhengjiManager: Manager<IRepository<...>>则更方便，因为子类方法中拿到的是具体的实现类型
